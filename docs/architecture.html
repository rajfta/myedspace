1. Purchase flow

Parent lands on the product page, buys a course, mock checkout runs, an invitation link is generated. No real payment is ever charged or stored.

GET /api/courses
Parent Browser
nginx
CoursesController
Public route. Renders the product page: Maths, English, Science — subject, year range, price.
POST /api/orders { courseId, parentEmail, studentEmail, card fields }
Parent Browser
OrdersController → OrdersService
Postgres
Card fields validated for shape only, never charged. Service creates an Order row, then a PENDING Enrollment row with a unique invitationToken.
200 OK { orderId, course, invitationToken, invitationPath }
Postgres
OrdersService
Parent Browser
Confirmation screen shows the invitation link directly (no email provider integrated) — /onboard/:invitationToken.

2. Student onboarding flow

Student opens the invitation link, sets a password (or, if that email already has an account, just confirms adding the new course to it), and receives a JWT.

GET /api/enrollments/:token
Student Browser
EnrollmentsController → EnrollmentsService
Postgres
Resolves the token. Three outcomes: invalid token → error page · already ACTIVATED → "already used, log in instead" · PENDING and email has no Student yet → full form · PENDING and email already has a Student → simplified "Welcome back!" panel (multi-course reuse).
POST /api/enrollments/:token/activate { fullName?, password? }
Student Browser
EnrollmentsService
Postgres
New email → creates Student (bcrypt-hashed password) and links it. Existing email → links the new Enrollment to the existing Student, no fields required. Either way, Enrollment.status → ACTIVATED.
200 OK { accessToken, student }
Postgres
AuthService (JWT sign)
Student Browser
Token stored client-side, browser redirected straight into /lms.

3. LMS access flow

Every route below is guarded by a Passport JWT strategy. No token, expired token, or a course the student never purchased all fail closed.

GET /api/me · GET /api/courses/:id/lessons · GET /api/lessons/:id   Authorization: Bearer <jwt>
Student Browser
JwtAuthGuard → LmsController
Guard rejects with 401 before the controller runs if the token is missing/invalid — confirmed live: clearing localStorage and reloading redirects to /login without ever reaching the API.
enrollment ownership check
LmsService
Postgres
Looks up the student's ACTIVATED enrollments. Course/lesson access is checked against that set — a course never purchased returns 403 ("You do not have access to this course."), verified live.
200 OK { dashboard | lessons[] | lesson }
Postgres
LmsService
Student Browser
Dashboard lists enrolled courses; lesson list shows per-lesson completion; POST /api/lessons/:id/complete writes a LessonProgress row, persisted server-side (survives a hard reload).

Data model

Prisma schema, Postgres. Enrollment is the join between a purchase and a student — a Student can hold multiple enrollments (buying a second course for an already-activated email reuses the account).

Course
  • id uuid
  • subject string
  • yearRange string
  • price int
  • createdAt datetime
Order
  • id uuid
  • courseId fk
  • parentEmail string
  • studentEmail string
  • amount int
Enrollment
  • id uuid
  • orderId fk, unique
  • courseId fk
  • studentEmail string
  • invitationToken uuid, unique
  • status PENDING|ACTIVATED
  • studentId fk, nullable
  • activatedAt datetime?
Student
  • id uuid
  • email string, unique
  • fullName string
  • passwordHash string
Lesson
  • id uuid
  • courseId fk
  • title string
  • description string
  • content string
  • order int
LessonProgress
  • id uuid
  • studentId fk
  • lessonId fk
  • completedAt datetime
  • unique(studentId, lessonId)
Course 1 → N Ordera course can be purchased many times
Course 1 → N Enrollmenta course can have many enrollments
Course 1 → N Lessona course has a fixed lesson list
Order 1 → 1 Enrollmentevery order immediately produces exactly one pending enrollment
Student 1 → N Enrollmentoptional until activation — studentId is nullable, set on activate; this is how multi-course reuse works
Student 1 → N LessonProgresscompletion is tracked per student, not per enrollment
Lesson 1 → N LessonProgressjoin entity between Student and Lesson, unique per pair
